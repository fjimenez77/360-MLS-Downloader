## Active ADRs

### ADR-001 — Single hard-dependency policy: `requests` only

**Status:** ✅ Accepted
**Date:** Pre-v2.0.0 (project inception)
**Source:** `CONTRIBUTING.md` §Guidelines — _"No dependencies — the only external dependency is `requests`. Keep it that way unless absolutely necessary."_

**Context:** This is a personal archival CLI tool. Easy installation matters; heavy dep trees
break that. Bring-your-own-Python is the expected install path.

**Decision:** Only `requests` is a hard runtime dependency. `playwright` is the single
permitted exception, justified by Zillow's CAPTCHA flow needing a real browser; it is
**conditional** (Zillow-only) and **auto-installed on first use** by `mls360-menu.py`.

**Consequences:**
- New providers must not introduce new deps without explicit maintainer approval.
- No web framework, no async runtime, no database, no logging library, no test framework.

### ADR-002 — Human-in-the-loop CAPTCHA is load-bearing legal posture

**Status:** ✅ Accepted
**Date:** Pre-v2.0.0
**Source:** `CONTRIBUTING.md` §Scope (verbatim): _"The human-in-the-loop CAPTCHA step is
load-bearing for this project's legal posture and is not up for negotiation — PRs that
remove it will be closed."_

**Context:** Zillow's listing pages are CAPTCHA-protected. The tool launches a real browser
and **waits for the human user to solve the CAPTCHA before continuing**.

**Decision:** The human-CAPTCHA step is non-removable. Not behind a flag, not "optional", not
"performance-mode-only." PRs framed any of those ways will be closed.

**Consequences:**
- Tool is fundamentally interactive for Zillow (not headless-by-default).
- This is the bright line between "personal archival tool" and "automated scraping tool."

### ADR-003 — No third-party egress: no proxies, CAPTCHA-solver APIs, telemetry

**Status:** ✅ Accepted
**Date:** Pre-v2.0.0
**Source:** `CONTRIBUTING.md` §Scope: _"PRs that add network egress to third-party services
(proxy providers, CAPTCHA-solver APIs, telemetry) will also be closed without review."_

**Context:** Same legal-posture and trust reasoning as ADR-002. Network egress is limited to
the platform being downloaded from (Zillow, Ricoh360) plus the vendored CDN-free viewer.

**Decision:** Outbound network calls are limited to: the target listing platform, its CDN/S3,
and `pip`/`playwright install` for install-time dependency fetch. Anything else is rejected.

**Consequences:**
- No analytics, no crash reporting, no usage telemetry.
- No proxy support (would enable bypassing platform rate limits / IP bans).
- No CAPTCHA-solver integration (would defeat ADR-002).

### ADR-004 — Provider registry pattern for new platforms

**Status:** ✅ Accepted
**Date:** Pre-v2.0.0 (encoded in `providers/__init__.py`)
**Source:** `CONTRIBUTING.md` §Adding a New Provider + source code

**Context:** Multiple MLS platforms need supporting (Zillow, Ricoh360 today; potentially
more). Hard-coding platform logic into the core engine would bloat it.

**Decision:** Each platform is a separate module in `providers/{name}.py` exporting:
- `PROVIDER_NAME` and `DISPLAY_NAME` constants
- `detect(url)` — boolean URL match
- `extract_ids(url)` — pull IDs out of URL
- `fetch_tour_data(session, ids)` — raw fetch
- `parse_tour(raw_data)` — normalize to standard tour dict

Providers are registered in `providers/__init__.py` `PROVIDERS = [zillow, ricoh360]`.
`detect_provider(url)` iterates the list in order.

**Consequences:**
- Adding a platform = one new file + one line in registry.
- Core engine stays platform-agnostic.
- Provider order in registry matters — earlier providers win ties.

### ADR-005 — Vendor Pannellum locally (no CDN)

**Status:** ✅ Accepted
**Date:** Pre-v2.0.0 (`vendor/pannellum.min.{css,js}` pinned 2026-03-22)
**Source:** Source code + `CONTRIBUTING.md` §Project Structure (_"vendor/ — Vendored Pannellum library (don't modify)"_)

**Context:** The HTML viewer needs a 360° panorama renderer. Pannellum (MIT) is the
standard choice. Loading it from a CDN would create an internet-dependency for an
otherwise-offline viewer.

**Decision:** Pannellum CSS+JS is vendored into `vendor/` and copied alongside the
generated HTML viewer. The viewer is fully offline.

**Consequences:**
- Larger viewer output (~66 KB per generated tour folder).
- Maintainer must manually bump Pannellum when needed (no auto-update).
- Aligns with ADR-003 (no third-party network egress).
